# 6. Repair refresh-token rotation and session security

The current refresh flow treats a legitimate replacement token as reused during the next refresh.

Implement a token-family model with correct replay detection.

## Required behavior

1. Accept the refresh token from a secure cookie.
2. Hash the presented token before database lookup.
3. Verify the token record is active, unexpired and not consumed or revoked.
4. Atomically mark the current token consumed.
5. Create one replacement token in the same family.
6. Store only the replacement token hash.
7. Return the raw replacement token only in the secure cookie.
8. Permit the replacement token to refresh normally.
9. Detect replay only when a consumed or revoked old token is presented again.
10. Revoke the entire token family on genuine replay.

Use one database transaction.

Also:

* Remove plaintext refresh-token storage.
* Remove known fallback JWT secrets.
* Fail startup if required secrets are missing.
* Make authentication cookies `httpOnly` where possible.
* Use `secure` in production and appropriate `sameSite` values.
* Revoke sessions on logout.
* Revoke all token families after password change.
* Revoke or invalidate sessions when a Student is blocked, rejected, deactivated or deleted.
* Add a single-flight client refresh mechanism so simultaneous `401` responses cause one refresh request.
* Retry each failed request at most once.
* Prevent infinite refresh loops.

Add tests for first refresh, second refresh, parallel refresh, expiry, replay, logout, password change and status revocation.
