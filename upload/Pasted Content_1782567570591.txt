======================================================================
1. PRIMARY PROJECT OBJECTIVE
======================================================================

Build a complete web-based education platform that allows an educational institute to:

1. Publish a professional public website.
2. register students.
3. Approve or reject student registrations.
4. Create and manage batches.
5. Assign students to one or multiple batches.
6. Organize courses, chapters, topics, and video lectures.
7. Deliver YouTube unlisted video content inside the application.
8. Upload batch-specific notes, assignments, test papers, and PDFs.
9. Create timed MCQ tests.
10. Restrict test attempts.
11. Automatically evaluate tests.
12. Track student results and learning progress.
13. Generate leaderboards and reports.
14. Display announcements.
15. Track admin activities and system events.
16. Operate securely on mobile, tablet, laptop, and desktop.
17. Scale to support a growing number of students and batches.

The application will initially support two primary roles:

- ADMIN
- STUDENT

Use strict role-based access control throughout the frontend, backend, API, and database workflows.

======================================================================
2. NON-NEGOTIABLE IMPLEMENTATION RULES
======================================================================

Follow these rules throughout the development process:

1. Do not build the complete application in a single file.
2. Do not mix frontend, backend, database, and configuration logic.
3. Do not hardcode admin passwords, JWT secrets, database credentials,
   storage keys, email credentials, or production URLs.
4. Do not expose sensitive data in frontend bundles or API responses.
5. Do not rely only on frontend validation.
6. Every sensitive rule must also be enforced by the backend.
7. Never trust user-provided role, user ID, score, test duration,
   batch assignment, attempt count, or video completion values.
8. Do not expose correct MCQ answers before an attempt is submitted.
9. Do not calculate final test scores on the client.
10. Do not allow blocked, rejected, pending, or inactive students to
    access protected learning content.
11. Do not allow a student to access another student's profile,
    results, batches, materials, attempts, or activity.
12. Do not allow deleted or unpublished content to remain accessible
    through direct API requests.
13. All destructive actions must require confirmation.
14. All asynchronous actions must display loading, success, and error states.
15. Prevent duplicate form submission and duplicate database records.
16. Use reusable components, services, validation schemas, and error handlers.
17. Use database transactions for critical multi-step operations.
18. Create database migrations and seed scripts.
19. Include automated tests for critical workflows.
20. Do not claim that an embedded YouTube URL is impossible to discover.
    Browser-based video URLs may still be inspected by technically skilled
    users. Reduce exposure through server-controlled video IDs,
    privacy-enhanced embeds, referrer policies, access validation, and UI
    restrictions, but do not make technically false security claims.

14.2 Question Management

Each question must include:

- Question text
- Four or configurable answer options
- Exactly one correct answer for single-choice mode
- Optional explanation
- Marks
- Display order

Validate:

- Required question text
- Minimum two options
- Exactly one correct option
- No blank options
- No duplicate display order
- Total questions within limit

Do not send `isCorrect` fields to the student before submission.

14.3 Student Test Attempt Workflow

Before starting an attempt, validate:

- Student account is approved and active
- Student is assigned to an authorized batch
- Test is published
- Test is within the availability window
- Attempt limit is not exhausted
- No invalid active attempt exists
- Test contains valid questions

When a student starts:

- Create an attempt server-side
- Save authoritative start time
- Calculate authoritative expiry time
- Return questions without correct-answer metadata
- Display countdown timer
- Save selected answers during the attempt
- Restore an active attempt after refresh
- Prevent creation of multiple simultaneous attempts for the same test

When time expires:

- Client must auto-submit
- Server must independently enforce expiry
- Late submissions must be finalized according to server time
- The timer must not reset after browser refresh
- Closing and reopening the browser must not create additional time

On submission:

- Use a database transaction
- Lock or atomically update the attempt
- Prevent duplicate scoring
- Calculate score on the backend
- Store selected answers
- Store correct/incorrect status
- Store marks
- Store percentage
- Store time taken
- Store submission type:
  - MANUAL
  - AUTO_TIMEOUT
  - ADMIN_FINALIZED
- Increment attempt count only through valid attempt records
- Return result according to configured visibility rules

After the second valid attempt:

- Lock further attempts
- Continue showing previous results
- Clearly display “Attempt limit reached”

40. ACCEPTANCE CRITERIA
======================================================================

The application is not considered complete until all of the following are true:

1. A student can register successfully.
2. The registration appears in the admin panel.
3. The admin can approve the student.
4. The student receives the correct status/email notification.
5. The approved student can log in.
6. A pending, blocked, inactive, or rejected student cannot access learning content.
7. The admin can create a batch.
8. The admin can assign the student to the batch.
9. The admin can create a course and assign it to the batch.
10. The admin can create chapters/topics.
11. The admin can add and publish a YouTube video.
12. The student can view only authorized content.
13. Video progress is saved and restored.
14. The admin can upload a PDF material.
15. The student can download only authorized material.
16. The admin can create a timed MCQ test.
17. The test rejects more than 20 questions.
18. The student can attempt the test only during the allowed window.
19. The timer remains valid after page refresh.
20. The test auto-submits when time expires.
21. Scoring occurs only on the backend.
22. The student cannot attempt the test more than twice.
23. The result displays correct score and answer comparison.
24. The admin can view all attempts and analytics.
25. Leaderboard updates after valid results.
26. Reports export successfully.
27. Announcements appear for the correct audience.
28. Contact messages reach the admin system.
29. Feedback appears in the admin panel.
30. Admin actions create audit-log entries.
31. Mobile and desktop interfaces work correctly.
32. Unauthorized direct API calls are rejected.
33. Upload validation works.
34. Rate limiting works.
35. Refresh-token rotation and logout work.
36. No secrets appear in the frontend or repository.
37. Database migrations work on a clean environment.
38. Docker-based local setup works.
39. Production build completes without type or lint errors.
40. Documentation allows another developer to run the system.
