   197‚Üí# 6. Repair refresh-token rotation and session security
   198‚Üí
   199‚ÜíThe current refresh flow treats a legitimate replacement token as reused during the next refresh.
   200‚Üí
   201‚ÜíImplement a token-family model with correct replay detection.
   202‚Üí
   203‚Üí## Required behavior
   204‚Üí
   205‚Üí1. Accept the refresh token from a secure cookie.
   206‚Üí2. Hash the presented token before database lookup.
   207‚Üí3. Verify the token record is active, unexpired and not consumed or revoked.
   208‚Üí4. Atomically mark the current token consumed.
   209‚Üí5. Create one replacement token in the same family.
   210‚Üí6. Store only the replacement token hash.
   211‚Üí7. Return the raw replacement token only in the secure cookie.
   212‚Üí8. Permit the replacement token to refresh normally.
   213‚Üí9. Detect replay only when a consumed or revoked old token is presented again.
   214‚Üí10. Revoke the entire token family on genuine replay.
   215‚Üí
   216‚ÜíUse one database transaction.
   217‚Üí
   218‚ÜíAlso:
   219‚Üí
   220‚Üí* Remove plaintext refresh-token storage.
   221‚Üí* Remove known fallback JWT secrets.
   222‚Üí* Fail startup if required secrets are missing.
   223‚Üí* Make authentication cookies `httpOnly` where possible.
   224‚Üí* Use `secure` in production and appropriate `sameSite` values.
   225‚Üí* Revoke sessions on logout.
   226‚Üí* Revoke all token families after password change.
   227‚Üí* Revoke or invalidate sessions when a Student is blocked, rejected, deactivated or deleted.
   228‚Üí* Add a single-flight client refresh mechanism so simultaneous `401` responses cause one refresh request.
   229‚Üí* Retry each failed request at most once.
   230‚Üí* Prevent infinite refresh loops.
   231‚Üí
   232‚ÜíAdd tests for first refresh, second refresh, parallel refresh, expiry, replay, logout, password change and status revocation.
